# Admin Site Settings — Design

**Date:** 2026-05-21
**Status:** Approved (autonomous build — user instructed "select recommended until PR open")

## 1. Goal

Give the admin a single panel at `/admin/site-settings` that controls four whole-site defaults:

1. **Default palette** — applied to visitors with no palette cookie.
2. **Default locale** — applied when a bare URL (no `/en|/ru|/be` prefix) is requested.
3. **Price overrides** — replace the base price of any service or membership tier.
4. **Global discount** — a percentage (0–90) with an on/off toggle that applies to every effective price across the site.

Per-user palette cookie and per-user locale URL prefix continue to take precedence over the admin defaults. The admin sets defaults, not hard locks.

## 2. Non-goals

- No per-user price overrides, no scheduled discounts, no coupon codes.
- No service catalog editing (names, blurbs, durations stay in `STUDIO_DATA`). Only the *price* of an existing catalog item is overridable.
- No multi-currency. Prices remain euros.
- No staff/role management beyond the existing `admin` role.

## 3. Storage

New singleton table:

```sql
CREATE TABLE site_settings (
  id                text PRIMARY KEY DEFAULT 'singleton'
                       CHECK (id = 'singleton'),
  default_palette   text NOT NULL DEFAULT 'aubergine',
  default_locale    text NOT NULL DEFAULT 'en',
  price_overrides   jsonb NOT NULL DEFAULT '{}'::jsonb,
  discount_percent  integer NOT NULL DEFAULT 0
                       CHECK (discount_percent BETWEEN 0 AND 90),
  discount_active   boolean NOT NULL DEFAULT false,
  updated_at        timestamptz NOT NULL DEFAULT now(),
  updated_by        text REFERENCES users(id)
);
```

`price_overrides` is a flat map keyed `"service:<id>"` or `"membership:<tier>"` → integer euros. Unknown keys are tolerated and ignored on read (forward-compatible with catalog renames).

The singleton `CHECK` constraint guarantees at most one row. Reads use a `getSiteSettings()` helper that lazily inserts the default row on first call and returns the in-memory defaults if `db` is null (graceful degradation, same pattern as the rest of `db/`).

## 5. Public APIs

### `entities/site-settings`

```ts
export interface SiteSettings {
  defaultPalette: PaletteId;
  defaultLocale: Locale;
  priceOverrides: Readonly<Record<string, number>>;
  discountPercent: number;     // 0..90
  discountActive: boolean;
  updatedAt: string;           // ISO
}

export const DEFAULT_SITE_SETTINGS: SiteSettings;

export type PriceOverrideKey =
  | `service:${string}`
  | `membership:${'Member' | 'VIP'}`;

export interface ResolvedPrice {
  base: number;       // catalog price (or override) before discount
  effective: number;  // base * (1 - discount/100), rounded
  hasDiscount: boolean;
}

export function resolvePrice(
  key: PriceOverrideKey,
  catalogPrice: number,
  settings: SiteSettings,
): ResolvedPrice;
```

`resolvePrice` is pure (no I/O), unit-tested exhaustively, and is the **only** path the rest of the app uses to render a price. Rounding rule: `Math.round` to whole euros — matches how prices are currently displayed (`€95`, not `€95.00`).

### `db/site-settings.ts`

```ts
export async function getSiteSettings(): Promise<SiteSettings>;
export async function updateSiteSettings(
  patch: Partial<Omit<SiteSettings, 'updatedAt'>>,
  updatedBy: string,
): Promise<SiteSettings>;
```

- `getSiteSettings()` returns the singleton row, lazily inserting if absent. Returns `DEFAULT_SITE_SETTINGS` (frozen object) when `db` is null.
- `updateSiteSettings()` validates the patch with a Zod schema before writing (palette must be a known `PaletteId`, locale must be in `routing.locales`, override keys must validate, discount in 0..90). Throws on validation failure — the server action catches it and re-renders the form with errors.

### `shared/lib/site-settings-server.ts`

```ts
import { cache } from 'react';
export const getSiteSettingsServer = cache(getSiteSettings);
```

Wrapped in React's `cache()` so the layout, proxy, and page can all call it once per request without duplicating DB hits.

## 6. How each setting is applied

### 6.1 Default palette

`app/[locale]/layout.tsx` currently hard-codes:

```tsx
<html data-palette={DEFAULT_PALETTE_ID} …>
```

Replaced with:

```tsx
const settings = await getSiteSettingsServer();
…
<html data-palette={settings.defaultPalette} …>
```

The inline `PALETTE_INIT_SCRIPT` already swaps in the user's cookie value if present, so per-user choice still wins. No client-side change.

### 6.2 Default locale

`i18n/routing.ts` exports `defineRouting({ ..., defaultLocale: 'en' })`. The default is consumed by next-intl's middleware (via `createMiddleware`) to decide where `/` redirects.

`createMiddleware` reads its config once per process boot. To respect a live admin choice without rebuilding, `proxy.ts` becomes:

```ts
import createMiddleware from 'next-intl/middleware';
import { routing } from './i18n/routing';
import { getSiteSettingsServer } from '@/shared/lib/site-settings-server';

export default async function proxy(req: NextRequest) {
  const settings = await getSiteSettingsServer();
  const handler = createMiddleware({
    ...routing,
    defaultLocale: settings.defaultLocale,
  });
  return handler(req);
}
```

The middleware is rebuilt per request; that's the recommended pattern from next-intl for runtime-derived defaults. (Cost is one DB read per request, mitigated by React `cache()` and the singleton row being trivially small. If profiling later shows it matters, we can add an in-memory TTL.)

### 6.3 Price overrides + discount

Every existing price render site reads through `resolvePrice`:

| File | Today | After |
|---|---|---|
| `entities/service/ui/service-card.tsx` | `€{service.price}` | `<Price resolved={resolvePrice('service:'+service.id, service.price, settings)} />` |
| `entities/service/ui/service-menu-item.tsx` | `€{service.price}` | same |
| `views/membership/ui/membership-page-client.tsx` `pricing()` helper | `tier.price * 10` etc. | reads resolved price |
| `views/service-detail/ui/sections/sticky-cta.tsx` | `price` prop (number) | accepts `ResolvedPrice` |
| `views/booking/ui/steps/confirm-step.tsx` & `confirmation-page.tsx` | `service.price` | resolved version |

`settings` is fetched once per server render at the page root and threaded down. Client components that need it receive it as a serialized prop (`SiteSettings` is JSON-safe).

A small new `<Price>` primitive in `shared/ui/price/` renders `<span>€{effective}</span>` plus a struck `<s>€{base}</s>` when `hasDiscount`. It's the only place that knows how a discounted price looks.

### 6.4 Discount on the booking confirmation

The booking flow already shows a single price line at confirmation. It will switch to the resolved price; the persisted `bookings.notes` is unaffected (we don't snapshot price into the DB today, and that's out of scope).

## 7. Admin UI

`/admin/site-settings` is a new page reachable from the admin dashboard tile grid. Layout:

1. **Header** — `AppHeader back="/admin" title={t('plate_title')}`.
2. **Form** (one server action, one submit button at the bottom):
   - Default palette — 12 radio chips, same UI as the existing per-user `PaletteSwitcher` but writing to the form, not the cookie.
   - Default locale — 3 radio chips (`en`, `ru`, `be`).
   - Service prices — table: name · category · base price · override input (number; empty = no override).
   - Membership tier prices — same shape for Member/VIP.
   - Global discount — number input (0–90) + active switch.
3. **Save** — submits the server action; on success, re-fetches and shows a small "Saved · just now" timestamp under the button.

Validation is server-side; client just clamps the discount input to 0–90 for UX.

A new `Admin.site_settings_*` translation namespace covers labels in all three locales.

## 8. Auth

The existing admin gate in `/admin` (TELEGRAM_BOT_TOKEN env var → session check → role check) extends to `/admin/site-settings`. The role check is the gate that's actually load-bearing here — only `users.role = 'admin'` can save. A new helper `requireAdmin()` in `shared/lib/auth-server.ts` centralizes the check so the page and the server action share it.

The server action also re-checks the session inside its body (defense in depth — server actions are externally callable endpoints).

## 9. Error handling

| Error | Behavior |
|---|---|
| `db` null (DB unconfigured) | `getSiteSettings()` returns frozen defaults; admin page shows a banner "Persistence disabled — saving is no-op" and disables the form. |
| Validation failure on save | Server action throws; page re-renders with the offending field highlighted and a translated error string. |
| Unknown override key on read | Silently ignored — the catalog item's base price is used. |
| Singleton race (two admins saving simultaneously) | Last write wins. The `updated_at` is shown in the UI so admins notice when someone else just edited. |

## 10. Testing

- **Unit (`vitest`)**: `resolvePrice` exhaustive table; `getSiteSettings` lazy-insert + db-null paths; Zod schema rejection cases.
- **Component (`vitest` + Testing Library)**: `SiteSettingsForm` renders fields, submits, surfaces validation errors. `<Price>` renders strikethrough only when `hasDiscount`.
- **Storybook**: stories for `SiteSettingsForm` (empty, populated, discount-active) and `<Price>` (no-discount, with-discount, free). All stories double as tests via `@storybook/addon-vitest`.
- **E2E (`playwright`)**: smoke test — admin logs in, changes palette + discount, verifies home page reflects new defaults for a fresh anonymous session. Skipped automatically when `TELEGRAM_BOT_TOKEN` is unset (matches the existing admin e2e pattern).

## 11. Migration

`npm run db:generate` produces `db/migrations/0005_*.sql` from the schema change. Backfill: `INSERT INTO site_settings (id) VALUES ('singleton') ON CONFLICT DO NOTHING;` runs as part of the same migration so production starts with the defaults populated.

## 12. Risks & open notes

- **Per-request middleware cost**: one DB select per request. Acceptable for a private-atelier site (low traffic); revisit with an LRU/TTL if traffic profile changes.
- **Price snapshotting at booking time**: out of scope. Today's `bookings` rows don't store the price paid — admin price changes therefore retroactively affect what the "Visits" view shows. Documented here so it's a known intentional limitation, not a bug.
- **Cookie palettes vs admin default**: anyone who has *ever* visited has a cookie, so the admin's choice mostly affects new visitors and incognito sessions. Acceptable; this is what "default" means.
