# PWA + Opt-in Web Push Notifications — Design

**Date:** 2026-05-24
**Status:** Approved (user fast-forwarded with "select recommended until PR open")
**Surface:** Site-wide service worker + `/[locale]/profile/notifications` settings page + 8 server-side trigger points

## 1. Problem & goals

Today the site is a regular Next.js 16 app. It already declares a Web App Manifest at [app/manifest.ts](../../../app/manifest.ts) but nothing else PWA — no service worker, no installability beyond what the manifest alone provides, no offline shell, no push channel. We want:

1. A real installable PWA — service worker, offline fallback, maskable icons, Lighthouse "installable" pass.
2. Opt-in browser notifications driven from the existing server actions, covering every meaningful state change in the app.
3. **All notifications off by default.** A signed-in user only gets notifications after explicitly enabling them in `/profile/notifications`.

## 2. Non-goals (YAGNI)

- Email, SMS, or Telegram-bot delivery channels. Web Push only in v1; the dispatcher is shaped so adding channels later is additive.
- Per-device toggling (a user with three devices either has all subscribed or none).
- Rich notification actions (action buttons, replies) — title + body + click-to-open URL is enough.
- A notifications inbox UI (in-app bell). Browsers already keep their own.
- A separate audit log table for notifications — `notification_log` doubles as that trail for v1.
- Background-sync, periodic-sync, badging API. Not worth the iOS-incompatibility surface.
- Reminder notifications further out than 24 h (no 1-week reminder).

## 3. Stack choices

- **Service worker tooling:** [`@serwist/next`](https://serwist.pages.dev/) — actively maintained Workbox successor with first-class Next.js App Router + Turbopack support. Replaces `next-pwa` (unmaintained, breaks on Next 16).
- **Push library:** [`web-push`](https://www.npmjs.com/package/web-push) — the reference Node implementation. Pure-JS, fine on Fluid Compute.
- **Crypto for VAPID keys:** generated once with `npx web-push generate-vapid-keys`, stored in env vars (see §10).

No Firebase, no APNs, no external services.

## 4. PWA pieces

### 4.1 Service worker
- New file [app/sw.ts](../../../app/sw.ts) (Next 16 App Router places SW source at app root; Serwist compiles it to `public/sw.js`).
- Precaches the build's static assets via Serwist's default Next.js manifest.
- Runtime caching:
  - **HTML** (navigation requests) → `NetworkFirst`, 3 s timeout, fall back to `/[locale]/offline` (server-rendered for `en`, `ru`, `by`).
  - **`/_next/static/**`** → `CacheFirst`, 30-day expiry.
  - **`https://*.public.blob.vercel-storage.com/**`** → `StaleWhileRevalidate`, 50-entry cap.
  - **Everything else (`/api/**`, auth routes, OG image, manifest, sitemap)** → `NetworkOnly`.
- `push` and `notificationclick` listeners (see §7.4).
- `next.config.ts` wraps the existing config with `withSerwist({ swSrc: "app/sw.ts", swDest: "public/sw.js", disable: process.env.NODE_ENV === "development" })`. Dev mode disables the SW so HMR isn't poisoned by cached chunks; we can flip it for manual testing.

### 4.2 Icons
- Add to `public/`:
  - `icon-192.png` — 192×192, opaque, `purpose: "any"`.
  - `icon-512.png` — 512×512, opaque, `purpose: "any"`.

  - `apple-touch-icon.png` — 180×180.

- Add `categories`, `lang: "en"`, `dir: "ltr"`, and one screenshot entry (`narrow` form factor) so Chrome shows a richer install prompt.

### 4.4 Service worker registration

## 5. Notification categories

Eight categories, two audiences. Wording lives in `messages/{en,ru,by}.json` under a new `Notifications` namespace. Both **title** and **body** are translated; the SW reads them from the push payload so the server picks the user's preferred locale (read from the `User.preferredLocale` — see §6.1).

| Audience  | Category id                  | Trigger                                                                    | Click target                          |
|-----------|------------------------------|----------------------------------------------------------------------------|---------------------------------------|
| Customer  | `booking_confirmed`          | `bookings.status` transitions to `confirmed`                               | `/[locale]/profile`                   |
| Customer  | `booking_cancelled`          | `bookings.status` transitions to `cancelled` (admin- or user-initiated)    | `/[locale]/profile`                   |
| Customer  | `booking_reminder_24h`       | cron — booking with `scheduled_for` 23–25 h out, status `confirmed`        | `/[locale]/profile`                   |
| Customer  | `vip_decision`               | `vip_requests.status` → `approved` or `declined`                           | `/[locale]/membership`                |
| Customer  | `testimonial_decision`       | `testimonials.status` → `approved` or `rejected`                           | `/[locale]/profile`                   |
| Admin     | `booking_created`            | new row in `bookings` (any status)                                         | `/[locale]/admin/bookings`            |
| Admin     | `vip_request_submitted`      | new row in `vip_requests` with `status='pending'`                          | `/[locale]/admin/vip-requests`        |
| Admin     | `testimonial_submitted`      | new row in `testimonials`, or change-request set on existing row           | `/[locale]/admin/testimonials`        |

Admin categories only render in the settings UI for users with `role='admin'`. Storing them on a customer is harmless but pointless — the dispatcher's role gate would skip them anyway.

## 6. Schema changes

One Drizzle migration: `db/migrations/0016_pwa_notifications.sql`.

### 6.1 `users.preferred_locale`
The notification dispatcher needs to know which locale to translate the payload into. The proxy already routes by URL locale, but a push notification is initiated server-side outside any request. Add:

```sql
ALTER TABLE users ADD COLUMN preferred_locale text NOT NULL DEFAULT 'en';
```

Set whenever the user signs in or changes their locale via the existing locale switcher. The user-facing payload uses this value; we fall back to `'en'` if it's somehow missing.

### 6.2 `notification_preferences`
```sql
CREATE TABLE notification_preferences (
  user_id text PRIMARY KEY REFERENCES users(id) ON DELETE CASCADE,
  categories jsonb NOT NULL DEFAULT '{}'::jsonb,
  updated_at timestamptz NOT NULL DEFAULT now()
);
```

`categories` shape: `Record<NotificationCategory, boolean>`. Missing keys are treated as `false` — this is what makes "off by default" hold without a backfill. When the user toggles an option the row is upserted with the full current map.

### 6.3 `push_subscriptions`
```sql
CREATE TABLE push_subscriptions (
  id text PRIMARY KEY,
  user_id text NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  endpoint text NOT NULL UNIQUE,
  p256dh text NOT NULL,
  auth text NOT NULL,
  user_agent text,
  created_at timestamptz NOT NULL DEFAULT now(),
  last_seen_at timestamptz NOT NULL DEFAULT now()
);
CREATE INDEX push_subscriptions_user_idx ON push_subscriptions (user_id);
```

`endpoint` is globally unique (it includes a browser-generated public key); the unique constraint also dedupes re-subscriptions from the same browser. On a 410/404 from `web-push` the dispatcher deletes the row.

### 6.4 `notification_log`
```sql
CREATE TABLE notification_log (
  id text PRIMARY KEY,
  user_id text NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  category text NOT NULL,
  payload jsonb NOT NULL,
  status text NOT NULL,         -- 'sent' | 'skipped_prefs' | 'no_subscriptions' | 'all_failed'
  error text,
  sent_at timestamptz NOT NULL DEFAULT now()
);
CREATE INDEX notification_log_user_idx ON notification_log (user_id, sent_at DESC);
CREATE INDEX notification_log_category_idx ON notification_log (category, sent_at DESC);
```

Not user-facing in v1 — only used for debugging and idempotency (the reminder cron checks this table to avoid double-sending).

### 6.5 Drizzle schema
Mirror the four tables in [db/schema.ts](../../../db/schema.ts) and export inferred types. `categories` typed as `jsonb<Partial<Record<NotificationCategory, boolean>>>`.

## 7. Server architecture

### 7.1 `shared/lib/notifications/`
New FSD-style shared module. Public API:

```ts
// shared/lib/notifications/index.ts
export { dispatchNotification } from "./dispatch";
export type { NotificationCategory, NotificationPayload } from "./types";
```

### 7.2 `dispatchNotification(userId, category, payload)`
```ts
async function dispatchNotification(
  userId: string,
  category: NotificationCategory,
  payload: { titleKey: string; bodyKey: string; bodyParams?: Record<string, string | number>; url: string },
): Promise<void>;
```

Steps, in order:
1. Read user + preferences in one query (left-join). If user missing → log `skipped_prefs`, return.
2. Resolve `enabled = preferences.categories[category] ?? false`. If `false` → log `skipped_prefs`, return.
3. Audience gate: admin-only categories require `user.role === 'admin'`. If not → log `skipped_prefs`, return.
4. Load `push_subscriptions` for user. If empty → log `no_subscriptions`, return.
5. Translate `titleKey`/`bodyKey` via `getTranslations({ locale: user.preferredLocale, namespace: 'Notifications' })`. Interpolate `bodyParams`. Result is the visible string for the browser.
6. Fan out — for each subscription, send via `web-push.sendNotification`. On `410 Gone`/`404` delete the subscription row. On other errors keep the subscription but record the error string. Update `last_seen_at` on success.
7. Write one `notification_log` row summarizing the fan-out (status = `sent` if ≥1 succeeded, `all_failed` otherwise).

The function is **fire-and-forget** from the caller's perspective: server actions `await` it (so errors surface in logs) but never throw on its failure — wrap the body in a try/catch and console.error. Notifications must never break a booking flow.

### 7.3 Trigger sites
A short audit per category. Each is a one-line `await dispatchNotification(...)` added after the existing DB write inside the matching server action — no new endpoints.

| Category                  | File                                                                                              | Insert point                                       |
|---------------------------|---------------------------------------------------------------------------------------------------|----------------------------------------------------|
| `booking_created`         | wherever new bookings land — check [features/booking](../../../features) and [app/api/booking](../../../app/api/booking) | after booking insert; recipient = each admin user  |
| `booking_confirmed`       | booking status mutator (same module)                                                              | after status update; recipient = booking.userId    |
| `booking_cancelled`       | [features/booking-cancel](../../../features/booking-cancel)                                       | after status update; recipient = booking.userId    |
| `booking_reminder_24h`    | new cron at `app/api/cron/booking-reminders/route.ts`                                              | iterate; recipient = booking.userId                |
| `vip_decision`            | [features/vip-requests-admin](../../../features/vip-requests-admin)                               | after status update; recipient = vipRequest.userId |
| `vip_request_submitted`   | [features/vip-request-submit](../../../features/vip-request-submit)                               | after insert; recipient = each admin user          |
| `testimonial_decision`    | [features/testimonials-admin](../../../features/testimonials-admin)                               | after status update; recipient = testimonial.userId|
| `testimonial_submitted`   | [features/testimonial-submit](../../../features/testimonial-submit)                               | after insert; recipient = each admin user          |

"Each admin user" = `SELECT id FROM users WHERE role='admin'`. Fan-out is sequential (admin count is small).

### 7.4 Service worker push handler
```ts
self.addEventListener("push", (event) => {
  const data = event.data?.json();
  event.waitUntil(
    self.registration.showNotification(data.title, {
      body: data.body,
      icon: "/icon-192.png",
      badge: "/icon-192.png",
      data: { url: data.url },
      tag: data.tag,           // category id — collapses duplicates
    }),
  );
});

self.addEventListener("notificationclick", (event) => {
  event.notification.close();
  event.waitUntil(self.clients.openWindow(event.notification.data.url));
});
```

Tag = category id so a second "booking_confirmed" replaces the first instead of stacking.

### 7.5 Cron endpoint
- File: `app/api/cron/booking-reminders/route.ts`.
- Trigger: `vercel.json` cron `0 * * * *` (hourly).
- Auth: `Authorization: Bearer ${process.env.CRON_SECRET}` (Vercel sets this automatically; we verify it).
- Query: `bookings WHERE status='confirmed' AND scheduled_for BETWEEN now()+'23 hours' AND now()+'25 hours' AND id NOT IN (SELECT (payload->>'bookingId') FROM notification_log WHERE category='booking_reminder_24h' AND sent_at > now()-'48 hours')`. (The dedup is what keeps an hourly cron from sending two reminders.)
- Calls `dispatchNotification` per match.

## 8. Client architecture

### 8.1 Settings page — `/[locale]/profile/notifications`
- New FSD slice `features/notification-preferences/`:
  - `ui/notification-settings.tsx` — toggle list + master "Enable browser notifications" button.
  - `ui/enable-push-button.tsx` — handles permission request + subscribe.
  - `model/use-push-subscription.ts` — Zustand store + actions: `subscribe()`, `unsubscribe()`, idle/subscribed/denied/unsupported state machine.
  - `api/server-actions.ts` — `saveSubscription`, `removeSubscription`, `updatePreferences`.
  - `index.ts` — public API.
- New view [views/profile-notifications/](../../../views/profile-notifications/) composes the feature.
- New page [app/[locale]/profile/notifications/page.tsx](../../../app/[locale]/profile/notifications/page.tsx).
- Entry tile on [views/profile](../../../views/profile) linking to the settings page.

### 8.2 Flow
1. User lands on the page. Toggles are visible but disabled until a subscription exists for this browser.
2. Master button reads `Notification.permission`:
   - `default` → request permission → if granted, subscribe via `PushManager.subscribe({ userVisibleOnly: true, applicationServerKey: vapidPublic })`, POST to `saveSubscription`. Toggles unlock.
   - `granted` but no DB row → silently re-subscribe.
   - `denied` → show inline guidance ("re-enable in browser settings"); toggles stay disabled.
   - Unsupported (no `PushManager`) → show inline message; UI fully disabled.
3. Toggles call `updatePreferences` server action on change (debounced 300 ms client-side to batch rapid taps).
4. "Disable" link calls `unsubscribe` server action + `PushSubscription.unsubscribe()`.

### 8.3 Locale capture
Add a one-liner to the existing locale switcher feature that POSTs to a new `saveLocalePreference` server action on switch (signed-in users only). Updates `users.preferred_locale`. No UI change.

## 10. Environment variables

Added to `.env.example`:

```dotenv
# --- Web Push ---
# Generated once with `npx web-push generate-vapid-keys`.
VAPID_PUBLIC_KEY=""
VAPID_PRIVATE_KEY=""
# Contact URI for push services (per RFC 8292). `mailto:` is conventional.
VAPID_SUBJECT="mailto:hello@violetta.example.com"
# Public copy of the VAPID public key — the browser needs this to
# call PushManager.subscribe. Identical value to VAPID_PUBLIC_KEY.
NEXT_PUBLIC_VAPID_PUBLIC_KEY=""

# --- Cron ---
# Vercel sets this automatically; required for self-host.
CRON_SECRET=""
```

Both `VAPID_PUBLIC_KEY` and `NEXT_PUBLIC_VAPID_PUBLIC_KEY` carry the same value — the duplication is deliberate so the client never reaches for a non-`NEXT_PUBLIC_*` env var.

## 12. Rollout plan

1. One migration, three new tables + one column. Default-off semantics mean zero behavioral change for existing users; no backfill.
2. Service worker ships disabled in dev. First production deploy registers the SW; old open tabs continue running without it until reload (acceptable).
3. After deploy: admin generates VAPID keys, sets env vars in Vercel, triggers redeploy. (The settings page renders an inline warning if the public VAPID key is empty so misconfiguration is obvious.)
4. Smoke test: admin opts in to one customer category, books a service on a second device, confirms in their inbox.

## 13. Risks & mitigations

| Risk                                                                     | Mitigation                                                                 |
|--------------------------------------------------------------------------|----------------------------------------------------------------------------|
| `web-push` throws and tanks a booking action                             | Dispatcher swallows errors; caller's try-catch on the action is unchanged. |
| User signs out / merges; orphan subscriptions                            | `ON DELETE CASCADE` from `users` covers it.                                |
| iOS users without an installed PWA never get push                         | Documented in the settings page intro copy; no engineering workaround.     |
| Cron double-fires reminders                                              | `notification_log` dedup query in §7.5.                                    |
| SW caches a stale `/sw.js` and never updates                             | Serwist sets `clientsClaim` + `skipWaiting`; manifest hash busts the cache.|
| Notifications shipped enabled accidentally                               | Default `'{}'::jsonb` in the migration + the dispatcher's `?? false` gate. |
| Per-action latency rises (extra DB calls per dispatch)                   | Reads are cheap (PK lookups); fan-out is bounded by admin/subscription count. If needed, move dispatch behind a Vercel Queue post-merge. |

## 14. Out of scope, captured for later

- Email/Telegram/SMS channels — additive change to `dispatchNotification`.
- Per-device unsubscribe — the schema supports it; just no UI today.
- In-app inbox UI — would read from `notification_log`.
- Marketing campaigns (broadcast notifications outside the event triggers) — explicitly excluded; the categories enum is the whitelist.
